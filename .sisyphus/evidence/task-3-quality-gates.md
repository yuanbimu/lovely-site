# 计划质量门槛清单

> **任务**: Task 3 - 项目计划质量阈值  
> **目的**: 定义可执行计划必须满足的最低结构性要求  
> **适用范围**: 所有 `.sisyphus/plans/*.md` 计划文档

---

## 0. 前置说明

**核心原则**: 代码写完 ≠ 任务完成。计划必须包含执行记录和验收记录，两者缺一不可。

| 概念 | 定义 | 示例 |
|------|------|------|
| 计划 (Plan) | 原始规划文档，包含目标、任务、验收条件 | TL;DR、Work Objectives、TODOs |
| 执行记录 (Execution Record) | 实际执行过程中产出的证据 | 代码路径、修改内容、完成时间 |
| 验收记录 (Verification Record) | 验证结果，证明验收条件已满足 | 测试截图、API响应、结构验证 |

**状态语义规则**:
- `TODO in_progress`: 正在执行，尚未完成
- `TODO completed`: 执行已完成，但未经验收
- `Acceptance Criteria checked`: 已通过验收

禁止出现：TODO 已标记完成但 Acceptance Criteria 仍有未完成项且无解释。

---

## 1. 目标 (Objectives)

### 强制要求

- [ ] TL;DR 包含：快速 Summary、交付成果、预计工作、并行执行策略
- [ ] 目标描述具体可量化，避免模糊形容词
- [ ] 明确核心目标和次要目标的区分

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| TL;DR 完整性 | 包含上述 4 个要素 |
| 目标可验证 | 目标可通过外部证据验证，而非主观判断 |
| 范围边界清晰 | 能判断某功能是否在范围内 |

### 反例

```markdown
# ❌ 不合格示例
目标：优化用户体验

# ✅ 合格示例
目标：在 AboutSection 添加两个按钮，实现橱窗选择和随机展示功能
```

---

## 2. 范围 (Scope)

### 强制要求

- [ ] Must Have: 必须实现的功能列表
- [ ] Must NOT Have: 明确排除的功能列表
- [ ] 范围边界可验证

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| Must Have 非空 | 至少列出 1 项必须实现的功能 |
| Must NOT Have 非空 | 至少列出 1 项明确不做的功能 |
| 边界可验证 | 通过代码路径或 API 可判断是否在范围内 |

---

## 3. 任务 (TODOs)

### 强制要求

- [ ] 每个 TODO 包含：What to do、Must NOT do、Acceptance Criteria
- [ ] 任务描述具体到文件路径和具体操作
- [ ] 明确推荐执行者类型（Recommended Agent Profile）

### 任务描述结构

```markdown
- [ ] N. 任务标题

  **What to do**:
  - 具体操作步骤 1
  - 具体操作步骤 2

  **Must NOT do**:
  - 明确禁止的操作

  **Recommended Agent Profile**:
  - Category: xxx
    - Reason: xxx

  **Acceptance Criteria**:
  - [ ] 验收条件 1
  - [ ] 验收条件 2
```

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| What to do 具体 | 能从描述直接执行，无需额外推理 |
| Must NOT do 明确 | 至少列出 1 项禁止项 |
| Acceptance Criteria 可验证 | 可通过工具验证，而非人工主观判断 |
| 无模糊完成表述 | 不出现"功能正常"、"验证通过"、"已修改" |

---

## 4. 依赖与并行 (Dependencies)

### 强制要求

- [ ] 任务间存在依赖时，明确标注阻塞关系
- [ ] 标注哪些任务可并行执行
- [ ] 标注并行执行的前提条件

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| 依赖关系明确 | 依赖任务明确列出被依赖的任务 |
| 并行策略合理 | 并行任务之间无数据依赖或资源冲突 |

### 示例

```markdown
**Parallelization**:
- **Can Run In Parallel**: YES
- **Blocks**: None
```

---

## 5. 验收条件 (Acceptance Criteria)

### 强制要求

- [ ] 每个任务必须有明确的 Acceptance Criteria
- [ ] 验收条件可量化、可验证、可证据化
- [ ] 区分"已实施"和"已验收"两种状态

### 验收条件分类

| 类型 | 描述 | 验证方式 |
|------|------|----------|
| 代码验证 | 文件创建、路径存在、结构正确 | grep/ls/read |
| API 验证 | API 响应格式、数据正确性 | curl/请求 |
| UI 验证 | 页面渲染、交互正常 | playwright |
| 数据验证 | 数据完整性、一致性 | 数据库查询 |

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| 非空 | 每个 TODO 至少有 1 个 Acceptance Criteria |
| 可验证 | 可通过工具验证，无需人工判断 |
| 证据化 | 验收结果可产出证据文件 |

---

## 6. QA 场景 (QA Scenarios)

### 强制要求

- [ ] 复杂功能必须包含 QA Scenarios
- [ ] 每个 Scenario 包含：Scenario 名称、Tool、Preconditions、Steps、Expected Result
- [ ] Preconditions 可在测试前准备

### 场景结构

```markdown
Scenario: 场景名称
  Tool: playwright/curl/手动
  Preconditions: 前置条件
  Steps:
    1. 步骤 1
    2. 步骤 2
    3. 步骤 3
  Expected Result: 期望结果
```

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| 步骤可执行 | 执行者可根据步骤直接操作 |
| 期望结果明确 | 可判断通过或失败 |
| 证据产出 | 场景执行后产出证据文件 |

---

## 7. 提交策略 (Commit Strategy)

### 强制要求

- [ ] 计划完成后提供提交信息模板
- [ ] 提交信息使用简体中文
- [ ] 使用约定的前缀：feat: / fix: / chore: / refactor:

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| 前缀规范 | 使用 feat: fix: chore: refactor: 之一 |
| 中文信息 | 提交信息为简体中文 |
| 描述具体 | 信息描述具体变更内容 |

---

## 8. 风险与回滚 (Risk / Rollback)

### 强制要求

- [ ] 高风险任务必须包含风险评估
- [ ] 包含回滚方案或恢复步骤
- [ ] 标注需要额外验证的高风险操作

### 风险分类

| 风险级别 | 定义 | 要求 |
|----------|------|------|
| 高 | 不可逆操作、数据丢失风险 | 必须包含回滚方案 |
| 中 | 可恢复但需额外操作 | 包含恢复步骤 |
| 低 | 可忽略或影响极小 | 可选 |

### 验收门槛

| 检查项 | 合格标准 |
|--------|----------|
| 高风险有方案 | 高风险任务包含回滚方案 |
| 回滚可执行 | 回滚步骤具体可执行 |

## 质量门槛检查清单

| 类别 | 检查项 | 状态 |
|------|--------|------|
| 目标 | TL;DR 包含 4 个要素 | [ ] |
| 目标 | 目标可验证 | [ ] |
| 范围 | Must Have 非空 | [ ] |
| 范围 | Must NOT Have 非空 | [ ] |
| 任务 | What to do 具体 | [ ] |
| 任务 | Must NOT do 明确 | [ ] |
| 任务 | Acceptance Criteria 可验证 | [ ] |
| 依赖 | 依赖关系明确 | [ ] |
| 验收 | 非空且可验证 | [ ] |
| QA | 复杂功能有场景 | [ ] |
| 提交 | 前缀规范 | [ ] |
| 风险 | 高风险有回滚 | [ ] |

**通过门槛**: 全部检查项通过方可视为合格计划。
