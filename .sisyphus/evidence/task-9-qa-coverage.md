# Task 9 - QA 覆盖率规范（文档任务专用）

> **任务**：定义文档任务的 QA 场景覆盖率要求
> **日期**：2026-04-16

---

## 概述

本文档定义文档任务的 QA 场景最低覆盖率要求，包括路径类型要求和工具映射规范。适用于代码文档、API 文档、组件文档等各类文档任务的质量验证。

---

## 最低覆盖率要求

### 路径类型要求

每个复杂功能必须包含 **至少 2 个 QA 场景**：

| 功能类型 | Happy Path | Failure Path | 合计 |
|----------|-----------|--------------|------|
| 简单功能 | 1 | 0 | 1 |
| 复杂功能 | 1 | 1 | 2 |
| 关键功能 | 2 | 1 | 3 |

**说明**：
- **简单功能**：单一操作，无明显失败场景（如"文件存在性检查"）
- **复杂功能**：包含多种输入或状态，存在失败可能（如"配置验证"）
- **关键功能**：影响核心流程，需要更多验证（如"数据同步"）

### 功能复杂度判断标准

以下情况视为 **复杂功能**：

1. **多状态** - 功能涉及多种状态变化
   - 示例：加载状态、成功状态、错误状态

2. **多输入** - 功能接受多种输入
   - 示例：下拉选择、随机选择、API 获取

3. **外部依赖** - 功能依赖外部资源
   - 示例：依赖 API、依赖数据库、依赖文件系统

4. **用户交互** - 功能包含用户操作
   - 示例：按钮点击、表单提交、导航

以下情况视为 **关键功能**：

1. **数据关键** - 功能处理核心数据
   - 示例：用户信息、支付信息

2. **流程关键** - 功能影响核心流程
   - 示例：登录流程、数据同步

3. **展示关键** - 功能影响页面展示
   - 示例：首屏渲染、图片展示

## Happy Path 要求

### 定义

验证功能在 **正常条件下** 正确工作的场景。

### 编写要求

1. **输入正常** - 使用正常的输入数据
   - 示例：非空的 API 返回值、有效的配置值

2. **环境正常** - 满足正常运行条件
   - 示例：开发服务器运行、文件存在

3. **结果正常** - 验证预期的成功结果
   - 示例：输出正确、内容完整

### 示例

```markdown
**Happy Path - 组件文档存在性验证**:

- Tool: read
- Preconditions:
  - 文件 src/components/home/ShowcasePicker.tsx 存在
- Steps:
  1. 读取文件内容
  2. 检查 JSDoc 注释
- Assertions:
  - 文件包含 JSDoc 注释
  - 组件正确导出
- Evidence Path: .sisyphus/evidence/task-9-component-docs.md
```

---

## Failure Path 要求

### 定义

验证功能在 **异常条件下** 的处理能力的场景。

### 编写要求

1. **异常条件** - 设置具体的异常状态
   - 示例：空数据、网络错误、文件不存在

2. **处理验证** - 验证错误处理逻辑
   - 示例：显示错误提示、保持可用状态

3. **恢复验证** - 验证恢复能力（可选）
   - 示例：重试后恢复、切换后正常

### 示例

```markdown
**Failure Path - 空数据处理**:

- Tool: grep
- Preconditions:
  - 拦截 API /api/showcases 返回空数组
- Steps:
  1. 打开主页
  2. 检查组件状态
- Assertions:
  - 显示"暂无橱窗"提示
  - 按钮保持禁用状态
- Failure Indicators:
  - 页面崩溃
  - 控制台错误
- Evidence Path: .sisyphus/evidence/task-9-empty-data.md
```

## 工具映射

### 文档任务常用工具

文档任务（代码文档、配置文档、类型文档）推荐使用以下工具：

| 工具 | 用途 | 适用场景 |
|------|------|----------|
| `read` | 读取文件内容 | 验证文档存在性、验证内容完整性 |
| `grep` | 搜索内容 | 验证特定字段、验证配置值 |
| `glob` | 查找文件 | 验证文件结构、验证命名规范 |
| `lsp_diagnostics` | 代码诊断 | 验证类型安全、验证语法正确 |
| `playwright` | 浏览器交互 | 验证文档渲染、验证交互功能 |

### 任务类型与工具映射

| 任务类型 | 推荐工具 | 工具选择原因 |
|----------|---------|--------------|
| 文档存在性检查 | `glob` + `read` | 验证文件存在且内容正确 |
| 配置值验证 | `read` + `grep` | 验证配置格式和字段值 |
| 类型定义验证 | `read` + `lsp_diagnostics` | 验证类型安全 |
| 组件文档验证 | `read` | 验证 JSDoc 存在 |
| API 文档验证 | `grep` | 验证 Endpoint 定义 |
| 页面渲染验证 | `playwright` | 需要浏览器渲染 |

### 不推荐使用的工具

以下工具 **不推荐** 用于文档任务：

| 工具 | 不推荐原因 |
|------|------------|
| `playwright` | 仅用于需要浏览器操作的场景，文档任务通常不需要 |
| `lsp_symbols` | 用于代码结构分析，不适合文档验证 |
| `ast_grep` | 用于代码重构，不适合文档验证 |

## 覆盖率检查清单

编写 QA 场景后检查以下项目：

### 基础检查

- [ ] 至少包含 1 个 happy path
- [ ] Happy path 包含明确的断言
- [ ] 复杂功能包含失败 path
- [ ] 所有路径包含证据路径

### 工具检查

- [ ] Tool 选择与任务类型匹配
- [ ] 不使用不必要的工具（如文档任务使用 playwright）
- [ ] 工具切换有明确理由

### 断言检查

- [ ] 断言明确且可判断
- [ ] 无模糊断言（如"正常"）
- [ ] 断言与步骤对应

### 证据检查

- [ ] 证据路径具体且可访问
- [ ] 证据格式正确（如 .png 用于截图）
- [ ] 证据命名规范

## 场景数量建议

根据功能复杂度，建议的 QA 场景数量：

| 功能类型 | 最小场景数 | 推荐场景数 |
|----------|-----------|------------|
| 简单功能 | 1 | 2 |
| 复杂功能 | 2 | 3 |
| 关键功能 | 3 | 4 |

**注意**：场景数量不应少于最小要求，但也不宜过多（场景过多增加维护成本）。

## 常见错误

### 错误 1：只有 Happy Path

```markdown
❌ 错误示例：
- 只有 1 个场景
- 只验证正常情况
```

```markdown
✅ 正确示例：
- 复杂功能至少 2 个场景
- 包含 happy path 和 failure path
```

### 错误 2：Happy Path 过于简单

```markdown
❌ 错误示例：
- Tool: read
- Steps: 验证文件存在
- Assertions: 文件存在
```

```markdown
✅ 正确示例：
- Tool: read + grep
- Steps: 读取文件并搜索关键字段
- Assertions: 字段存在且值正确
```

### 错误 3：Failure Path 形同虚设

```markdown
❌ 错误示例：
**Failure Path - 空数据**:
- Preconditions: 无
- Steps: 检查是否正常
- Assertions: 正常
```

```markdown
✅ 正确示例：
**Failure Path - 空数据处理**:
- Preconditions: API 返回空数组
- Steps: 检查组件显示状态
- Assertions: 显示提示信息，按钮禁用
- Failure Indicators: 页面崩溃
```
