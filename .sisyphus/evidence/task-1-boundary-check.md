# 分类边界检查与反例验证

## 概述

本文档作为分类矩阵的补充，专门验证边界条件和易混淆场景。通过反例分析，确保分类标准的准确性和可操作性。

---

## 边界条件定义

### 边界条件 1：看起来像计划但不是

**特征**：文件包含「目标」「任务」「TODO」等关键词，但缺乏计划的核心要素。

**反例场景**：

1. **仅有任务列表，无验收标准**
   ```markdown
   # 本周工作
   - 完成用户登录功能
   - 修复首页样式问题
   - 优化数据库查询
   ```
   **分类**：草稿（缺少验收标准和执行策略）

2. **仅有目标声明，无任务拆分**
   ```markdown
   # 项目目标
   目标是重构整个后台管理系统，使其更加现代化。
   ```
   **分类**：其他或草稿（缺乏可执行的任务拆分）

3. **仅有 TODO 标记，无状态追踪**
   ```markdown
   # 开发计划
   - [ ] 添加用户认证
   - [ ] 实现数据导出
   - [ ] 优化性能
   ```
   **分类**：草稿（缺少验收标准和依赖关系）

---

### 边界条件 2：工具生成但包含分析

**特征**：文件由工具生成，但包含人工添加的分析或结论。

**反例场景**：

```markdown
# API 抓取结果

原始 JSON：
```json
{"success": true, "data": [...]}
```

**分析**：API 返回数据正常，但缺少错误处理机制。
```

**分类**：证据快照为主，人工分析部分可视为「其他」。整体以证据快照论处。

---

### 边界条件 3：混合类型的文件

**特征**：文件前半部分像计划，后半部分像快照。

**反例场景**：

```markdown
# 橱窗功能改进计划

## 目标
实现橱窗轮播功能

## 任务
- [x] 创建轮播组件
- [ ] 集成到页面

---

# 抓取结果
（后接 Playwright 输出）
```

**分类**：按主导内容判定。若计划部分占主导且结构完整，可视为正式计划；否则视为草稿或混合类型。

## 根目录 6 个文档的边界检查

### 检查方法：八项测试

对每个根目录文档应用以下 8 项测试，每项通过则得分：

| 序号 | 测试项 | 说明 |
|------|--------|------|
| 1 | 有 TL;DR 或执行摘要？ | 快速理解工作范围和交付物 |
| 2 | 有明确的「必须做」边界？ | 界定范围边界 |
| 3 | 有原子化任务拆分（TODO）？ | 工作可分解 |

| 7 | 有执行策略（分阶段/执行顺序）？ | 有实施路径 |
| 8 | 状态标记一致（打勾与内容匹配）？ | 进度可信 |

**正式计划门槛**：至少通过 6 项

## 关键发现

### 6 个根目录文档的共性特征

1. **格式均为 DOM 树或 JSON**：由 Playwright 抓取或 API 工具自动生成
2. **无目标声明**：不包含「要做...」「交付...」等表述
3. **无任务结构**：不包含 TODO 列表或任务拆分
4. **无验收标准**：不包含 Acceptance Criteria
5. **无验证策略**：不包含 QA 场景或验证命令
6. **无执行意图**：不描述任何未来工作

**结论**：这 6 个文档不符合「计划」的核心定义，即「对未来工作的规划」，而是「对现状的记录」。

## 分类决策验证流程

### Step 1：检查文件来源

```
文件是工具自动生成的吗？（Playwright/curl/API 客户端）
  ├─ 是 → 进入 Step 2
  └─ 否 → 进入 Step 3
```

### Step 2：验证是否为纯快照

```
文件仅包含原始输出（JSON/DOM/响应体）吗？
  ├─ 是 → 证据快照
  └─ 否 → 可能包含人工分析，按主导内容判定
```

### Step 3：检查计划核心要素

```
应用八项测试，得分 ≥ 6 → 正式计划
得分 3-5 → 草稿
得分 0-2 → 其他（或降级为证据快照）
```

## 结论

1. **根目录 6 个文档全部为证据快照**，八项测试得分均为 0，不具备任何计划属性
2. **分类矩阵标准清晰**，正式计划需满足至少 6 项八项测试
3. **边界条件处理原则**：当无法判定时，优先降级（正式计划 → 草稿 → 证据快照）而非升级
4. **后续行动**：建议将这 6 个证据快照移动至 `.sisyphus/evidence/` 目录，避免与计划混淆
