# 高风险操作边界定义

## 概述

本文档明确界定哪些操作被归类为高风险，以及对应的审批要求。

---

## 高风险操作分类

### A 类 - 数据库操作 (最高风险)

| 操作类型 | 风险等级 | 示例 |
|---------|---------|------|
| 数据删除 | 极高 | 删除用户数据、删除历史记录 |
| 数据批量更新 | 高 | 批量修改 image_url、更新用户权限 |
| 结构变更 | 极高 | 删除表、添加/删除列、修改索引 |
| 数据迁移 | 高 | 从一个表迁移到另一个表 |

**影响**: 不可逆的数据变更，可能导致数据永久丢失

---

### B 类 - R2 存储操作 (高风险)

| 操作类型 | 风险等级 | 示例 |
|---------|---------|------|
| 文件删除 | 高 | 删除 CDN 图片、删除归档文件 |
| 文件移动 | 高 | 目录重组、文件名变更 |
| 批量上传 | 中 | 批量上传占位图、批量迁移文件 |
| 权限变更 | 高 | 修改 bucket 访问权限 |

**影响**: 静态资源丢失或无法访问，影响前端显示

---

### C 类 - 远程 API 操作 (中风险)

| 操作类型 | 风险等级 | 示例 |
|---------|---------|------|
| 认证 API | 中 | 调用登录、获取 token |
| 第三方支付 | 高 | 支付接口、退款接口 |
| 数据同步 | 中 | 同步外部数据源 |
| Webhook | 中 | 接收外部回调 |

**影响**: 可能导致服务中断、资金损失、数据不一致

---

### D 类 - 生产环境操作 (最高风险)

| 操作类型 | 风险等级 | 示例 |
|---------|---------|------|
| 直接修改生产配置 | 极高 | 修改 wrangler.toml、生产环境变量 |
| 域名变更 | 极高 | 修改 DNS 记录、添加新域名 |
| 服务降级 | 高 | 关闭服务、限制访问 |
| 数据导出 | 中 | 导出用户数据、导出日志 |

**影响**: 服务中断、数据泄露、安全风险

---

## 风险判定流程

```
开始
  ↓
是否涉及数据库? → 是 → A类 (最高风险)
  ↓ 否
是否涉及 R2/存储? → 是 → B类 (高风险)
  ↓ 否
是否涉及远程 API? → 是 → C类 (中风险)
  ↓ 否
是否涉及生产环境? → 是 → D类 (最高风险)
  ↓ 否
→ 低风险操作 (无需额外审批)
```

| 风险类别 | 备份前提 | 干运行 | 批量分段 | 失败停止 | 恢复步骤 | 验证步骤 |
|---------|---------|--------|---------|---------|---------|---------|
| A 类 (数据库) | 必须 | 必须 | 必须 | 必须 | 必须 | 必须 |
| B 类 (R2) | 必须 | 必须 | 必须 | 必须 | 必须 | 必须 |
| C 类 (API) | 建议 | 必须 | 建议 | 必须 | 必须 | 必须 |
| D 类 (生产) | 必须 | 必须 | 必须 | 必须 | 必须 | 必须 |

## 反面教材 (admin-improvements.md 问题分析)

### 问题 1: 缺少备份前提

在 `admin-improvements.md` 计划中:
- **R2 图片迁移**: 计划直接从 showcase 复制到 model-3~32
- **数据库更新**: 计划直接修改 image_url 字段
- **问题**: 没有提及任何备份步骤

**正确做法**:
```markdown
### 备份前提
- [ ] 导出 showcases 表当前数据到 JSON
- [ ] 列出 R2 所有文件到 backup-r2-files.json
- [ ] 备份当前数据库到 backup-2024-01-01.sql
```

---

### 问题 2: 缺少批量分段

计划中提到:
- model-3~32 共 30 个目录需要上传占位图
- 数据库 30 条记录需要更新 image_url

**问题**: 计划未说明如何分批执行，假设一次性完成

**正确做法**:
```markdown
### 批量分段
- R2 上传: 每批 5 个目录，共 6 批
- 数据库更新: 每批 10 条记录，共 3 批
- 每批完成后检查成功/失败数量
```

---

### 问题 3: 缺少失败停止条件

计划中没有任何关于失败判定的说明

**正确做法**:
```markdown
### 失败停止条件
- 任何 R2 上传失败 → 停止并回滚
- 数据库更新失败 → 停止并回滚
- 成功率低于 95% → 停止并回滚
```

---

### 问题 4: 恢复步骤不具体

计划中提到"备份后删除"，但没有具体恢复步骤

**正确做法**:
```markdown
### 恢复步骤
1. 停止所有执行中的上传任务
2. 恢复 R2: 使用备份文件重新上传到原路径
3. 恢复数据库: 使用备份 SQL 执行更新
4. 验证: 检查 image_url 是否恢复到原始值
```

---

### 问题 5: 验证步骤不完整

计划中的验证命令:
- 访问 /admin 检查布局
- 访问 /showcase 检查轮播

**问题**: 没有验证数据完整性

**正确做法**:
```markdown
### 验证步骤
- [ ] 数据库记录数与备份前一致 (30 条)
- [ ] 所有 image_url 可正常访问
- [ ] R2 文件数量与备份前一致
- [ ] 前台页面显示正常
```

---

## 总结

高风险操作必须满足以下条件:

1. **明确风险类别**: 清楚属于 A/B/C/D 中的哪一类
2. **完整回滚计划**: 6 个强制字段全部填写
3. **具体可执行**: 每一步都有明确的操作说明
4. **可验证**: 有明确的验证标准和检查点
5. **无"尝试看看"**: 禁止"先做了再看"的心态
