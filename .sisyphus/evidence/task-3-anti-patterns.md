# 计划反模式清单

> **任务**: Task 3 - 常见反模式识别  
> **目的**: 列出使计划不可执行的常见错误模式  
> **适用范围**: 所有计划编写者和审查者

---

## 0. 概念区分

在讨论反模式之前，必须明确三个核心概念：

**计划 (Plan)**: 原始规划文档，说明要做什么、怎么做、验收标准是什么。包含 TL;DR、Work Objectives、TODOs、Verification Strategy 等部分。

**执行记录 (Execution Record)**: 执行过程中产出的实际证据。记录代码写在哪里、修改了什么、什么时候完成。例如："CSS 已修改，文件路径: src/components/AdminDashboard.css:96"

**验收记录 (Verification Record)**: 验证结果，证明验收条件已满足。例如："Playwright 测试通过，截图: .sisyphus/evidence/test-1.png"

三者关系：计划指导执行，执行产生记录，记录用于验收。没有执行记录的验收是空谈。

## 1. 验收条件类反模式

### 反模式 1.1：模糊验收表述

**典型表述**: "功能正常"、"验证通过"、"已修改"、"看起来没问题"

**问题分析**: 这类表述无法验证。什么是"功能正常"？谁来判断？通过什么工具判断？

**错误示例**:

```markdown
# ❌
Acceptance Criteria:
- [ ] 功能正常
- [ ] 验证通过
```

**正确示例**:

```markdown
# ✅
Acceptance Criteria:
- [ ] 访问 /admin 页面，内容区域居中显示（max-width: 1200px）
- [ ] 页面宽度超过 1000px 时显示 3 列网格
- [ ] Playwright 截图验证布局符合设计
```

### 反模式 1.2：验收条件与任务脱节

**典型表现**: 验收条件描述的内容与任务实际要做的事情不匹配。

**问题分析**: 任务说要做 A，验收条件却验证 B，导致执行者不知道做到什么程度算完成。

**错误示例**:

```markdown
# ❌ 任务说要修改 CSS，验收条件却验证 API
TODO 1: 修改 Admin 布局 CSS
  Acceptance Criteria:
  - [ ] /api/showcases 返回正确数据
```

**正确示例**:

```markdown
# ✅
TODO 1: 修改 Admin 布局 CSS
  Acceptance Criteria:
  - [ ] src/components/AdminDashboard.css 包含 max-width: 1200px
  - [ ] 访问 /admin 页面内容区域居中
```

### 反模式 1.3：验收条件无证据化

**典型表现**: 验收通过但无任何证据文件，无法事后追溯。

**问题分析**: 没有证据的验收等于没有验收，无法审计也无法复现。

**正确做法**: 验收条件完成后，产出证据到 `.sisyphus/evidence/` 目录。

## 2. 任务描述类反模式

### 反模式 2.1：任务描述过于笼统

**典型表述**: "优化用户体验"、"改进布局"、"完善功能"

**问题分析**: 执行者无法从描述中得知具体要做什么，需要额外推理或询问。

**错误示例**:

```markdown
# ❌
What to do:
- 优化 Admin 页面布局
```

**正确示例**:

```markdown
# ✅
What to do:
- 修改 src/components/AdminDashboard.css
- 内容区域添加 max-width: 1200px; margin: 0 auto;
- 卡片容器添加 grid-template-columns: repeat(3, 1fr);
```

### 反模式 2.2：缺少 Must NOT do

**典型表现**: 只告诉执行者要做什么，不告诉不要做什么。

**问题分析**: 执行者可能做超出范围的事情，或者做了不该做的操作。

**正确做法**: 每个任务必须列出至少一项 Must NOT do。

### 反模式 2.3：任务间无依赖说明

**典型表现**: 多个任务之间有依赖关系，但没有标注。

**问题分析**: 执行者可能并行执行依赖任务，导致失败或需要返工。

**正确做法**: 在任务中标注 Parallelization 信息。

## 3. 状态语义类反模式

### 反模式 3.1：代码写完等于任务完成

**典型表现**: TODO 标记为 completed，但 Acceptance Criteria 仍为空或未勾选。

**问题分析**: 代码写完只是"已实施"，不是"已完成"。没有验收的代码无法保证质量。

**正确语义**:
- `TODO in_progress`: 正在写代码
- `TODO completed`: 代码已写完，等待验收
- `Acceptance Criteria checked`: 验收通过

### 反模式 3.2：同时混用矛盾状态

**典型表现**: 同一个任务，既标记为已完成，又标记为待验收，且无解释。

**正确做法**: 状态必须一致。如有特殊情况，需在备注中说明原因。

## 4. QA 场景类反模式

### 反模式 4.1：QA 场景无法执行

**典型表现**: 步骤描述模糊，或者缺少必要的前置条件。

**错误示例**:

```markdown
# ❌
Scenario: 测试下拉菜单
  Steps:
    1. 打开页面
    2. 点击下拉菜单
    3. 验证功能正常
```

**正确示例**:

```markdown
# ✅
Scenario: 测试下拉菜单选择橱窗
  Tool: playwright
  Preconditions: 开发服务器已启动，页面可访问
  Steps:
    1. 打开 http://localhost:4321/
    2. 滚动到 About 区域
    3. 点击下拉选择框
    4. 选择第二个选项
    5. 检查图片 src 是否变化
  Expected Result: 图片 src 变为选中橱窗的 URL
```

### 反模式 4.2：期望结果无法判断

**典型表现**: 期望结果用主观描述，无法客观判断通过或失败。

**错误示例**:

```markdown
# ❌
Expected Result: 页面看起来正常
```

**正确示例**:

```markdown
# ✅
Expected Result: 按钮文字变为"已选择"，图片 src 包含 showcase-2
```

## 5. 范围边界类反模式

### 反模式 5.1：Must Have 与 Must NOT Have 混为一谈

**典型表现**: 范围描述混乱，该做的和不该做的没有明确区分。

**正确结构**:

```markdown
### Must Have
- 从 /api/showcases 获取数据
- 下拉选择框 + 随机按钮

### Must NOT Have
- 不创建新的 API 端点
- 不修改数据库
```

### 反模式 5.2：范围边界无法验证

**典型表现**: 说不在范围内，但无法通过代码路径或 API 验证。

**正确做法**: 范围边界必须可通过工具验证。

---

## 6. 综合反模式

### 反模式 6.1：计划与执行记录混淆

**典型表现**: 计划文档中直接写执行结果，没有区分计划阶段和执行阶段。

**正确做法**: 计划是规划文档，执行记录是产出证据。两者分开记录。

### 反模式 6.2：缺少风险评估

**典型表现**: 高风险操作（如数据库迁移）没有风险评估和回滚方案。

**正确做法**: 高风险任务必须包含风险级别和回滚步骤。

## 总结

计划反模式的核心问题是：**模糊**和**缺失**。

验收条件模糊 → 无法验证  
任务描述模糊 → 无法执行  
状态语义模糊 → 无法审计  

缺失执行记录 → 无法追溯  
缺失验收记录 → 无法证明  
缺失风险方案 → 无法兜底  

解决这些问题的方法：把一切都说清楚、写具体、可验证、能证据化。
